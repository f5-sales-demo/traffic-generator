Todos os arquivos Terraform estão no diretório [`terraform/`](https://github.com/f5xc-salesdemos/traffic-generator/tree/main/terraform). Clone o repositório e implante diretamente:

```bash
git clone https://github.com/f5xc-salesdemos/traffic-generator.git
cd traffic-generator/terraform
cp terraform.tfvars.example terraform.tfvars
# Edite terraform.tfvars com seu ID de assinatura Azure e FQDN de destino
```

## Configuração do Terraform

### Estrutura de Arquivos do Terraform

O diretório terraform contém 9 arquivos seguindo o Padrão de Recursos de Demonstração:

- `versions.tf` -- Restrições de versão do Terraform e provedores (azurerm ~> 4.0, azuread ~> 3.0)
- `providers.tf` -- Configuração dos provedores Azure RM e Azure AD
- `data.tf` -- Fontes de dados Azure AD para resolução automática do implantador
- `locals.tf` -- Resolução do implantador, nomenclatura de recursos Azure Cloud Adoption Framework, tags padrão
- `main.tf` -- Grupo de recursos (nomeado `rg-traffic-generator-{environment}-{deployer}`)
- `variables.tf` -- Todas as variáveis de entrada (2 obrigatórias, 9 opcionais) com resolução automática do implantador via Azure AD
- `network.tf` -- VNet (10.201.0.0/16), sub-rede, IP público, NSG (porta 22 SSH), NIC
- `vm.tf` -- VM Ubuntu 24.04 com cloud-init via templatefile()
- `outputs.tf` -- 17 saídas: 15 saídas padrão compartilhadas por todos os recursos de demonstração (deployer, public_ip, private_ip, ssh_command, resource_group_name, vm_name, nsg_name, vnet_name, subnet_id, component, environment, resource_group_id, vm_id, nsg_id, location) mais 2 saídas específicas do componente (target_fqdn, status_check)

### Provedor e Variáveis

`main.tf` configura o provedor Azure:

```hcl file=../_data/main.tf
```

`variables.tf` define todos os parâmetros configuráveis. O padrão de `vm_size` é `Standard_F16s_v2` (16 vCPU otimizado para computação) -- consulte [Dimensionamento da VM](#dimensionamento-da-vm) para orientações de escalabilidade:

```hcl file=../_data/variables.tf
```

### Infraestrutura de Rede

`network.tf` cria a VNet, sub-rede, IP público, NSG (porta 22 SSH) e NIC:

```hcl file=../_data/network.tf
```

### Máquina Virtual com Cloud-Init

`vm.tf` cria a VM Ubuntu 24.04 e passa o template cloud-init com variáveis de destino e ferramentas:

```hcl file=../_data/vm.tf
```

### Provisionamento Cloud-Init

`cloud-init.yaml` é a configuração validada em produção, verificada em mais de 48 horas de testes de carga contínuos, incluindo 4 rodadas de testes de estresse na origem. Utiliza uma biblioteca auxiliar com lógica de retry, registro de progresso em `/var/log/cloud-init-progress.log`, 9 fases de provisionamento e relatório condicional de status (ready/degraded em `status.json`). Provisiona:

- **Ajuste de kernel** -- `somaxconn=131072`, `tcp_max_tw_buckets=4000000`, `tcp_fin_timeout=5`, buffers de socket de 64 MiB, `file-max=4194304`
- **Otimização de NIC** -- RPS/RFS em todos os cores, buffers de anel maximizados via ethtool, THP=always
- **Pacotes APT** -- nikto, nmap, masscan, sqlmap, hydra, medusa, ncrack, tshark, hping3, tcpdump, netcat, ngrep, iperf3, mtr, sslscan, socat, dirb, whatweb, wrk, hey, vegeta, ethtool
- **Releases do GitHub** -- nuclei, dalfox, ffuf, gobuster, feroxbuster, subfinder, httpx, amass
- **Pacotes Python** -- mitmproxy, sslyze, dnsrecon, fierce, theHarvester, playwright, scapy, impacket, arjun
- **Node.js 20.x** -- Puppeteer com plugin stealth para simulação de bots
- **Automação de navegador** -- Playwright Chromium para testes de navegador headless
- **Clones Git** -- testssl.sh, recon-ng, spiderfoot, SecLists, waf-bypass, PayloadsAllTheThings
- **Implantação de suítes** -- 19 suítes de tráfego clonadas deste repositório em `/opt/traffic-generator/suites/`
- **Arquivo de configuração** -- `config.env` com `TARGET_FQDN`, `TARGET_ORIGIN_IP` e `CRAPI_PORT`

O cloud-init utiliza variáveis de template do Terraform: `${target_fqdn}`, `${target_origin_ip}` e `${tool_tier}`. Variáveis shell como `$NPROC` são escapadas como `$$NPROC` no templatefile do Terraform.

```yaml file=../_data/cloud-init.yaml
```

`outputs.tf` expõe 17 saídas incluindo IP público/privado, comando SSH, identificadores de recursos e valores específicos do componente (target_fqdn, status_check):

### Exemplo de Arquivo de Variáveis

Copie `terraform.tfvars.example` para `terraform.tfvars` e preencha seus valores. O `.gitignore` exclui `terraform.tfvars` para evitar o commit de credenciais:

```hcl file=../_data/terraform.tfvars.example
```

Substitua `subscription_id` pelo seu ID de assinatura Azure (obtido com `az account show --query id -o tsv`) e `target_fqdn` pelo domínio do seu balanceador de carga F5 XC.

## Implantar

```bash
terraform init

terraform plan

terraform apply
```

O Terraform exibe o IP público, comando SSH e FQDN de destino após a implantação bem-sucedida.

## Dimensionamento da VM

As VMs série F otimizadas para computação são recomendadas para esta carga de trabalho de geração de tráfego com uso intensivo de CPU. O ajuste de kernel do cloud-init e a concorrência das ferramentas escalam automaticamente com a quantidade de vCPUs.

| SKU da VM | vCPU | RAM | Rede | Caso de Uso |
|---|---|---|---|---|
| Standard_F4s_v2 | 4 | 8 GiB | 10 Gbps | Laboratório/demonstração leve |
| Standard_F8s_v2 | 8 | 16 GiB | 12.5 Gbps | Demonstrações padrão |
| Standard_F16s_v2 | 16 | 32 GiB | 12.5 Gbps | Testes de carga (padrão) |
| Standard_F32s_v2 | 32 | 64 GiB | 16 Gbps | Benchmarking pesado |
| Standard_D16s_v3 | 16 | 64 GiB | 12.5 Gbps | Cargas com uso intensivo de RAM |

:::tip
F16s_v2 foi validado como a opção ideal de custo-benefício por meio de testes de benchmark A/B/C/D contra alvos CDN F16s_v2 e F32s_v2. Consulte [benchmarks/](https://github.com/f5xc-salesdemos/traffic-generator/tree/main/benchmarks) para os resultados completos.
:::

## Verificação Pós-Implantação

O provisionamento do cloud-init leva **15-20 minutos para o tier padrão** e **aproximadamente 25 minutos para o tier completo**. A VM fica acessível via SSH imediatamente, mas as ferramentas ainda estão sendo instaladas até que o cloud-init seja concluído.

### Aguardar o Cloud-Init

Conecte-se à VM via SSH e aguarde o provisionamento finalizar:

```bash
ssh azureuser@$(terraform output -raw public_ip)

cloud-init status --wait
```

Saída esperada quando concluído:

```
status: done
```

### Verificar Arquivo de Status

O script cloud-init grava um arquivo de status ao concluir:

```bash
cat /opt/traffic-generator/status.json
```

Saída esperada:

```json
{
  "status": "ready",
  "tier": "standard",
  "target_fqdn": "demo.example.com",
  "completed": "2026-04-25T14:30:00Z",
  "tools_installed": true
}
```

### Verificar Ferramentas Críticas

Execute um loop de verificação para confirmar que cada ferramenta crítica está disponível no PATH:

```bash
for tool in nikto sqlmap nuclei dalfox ffuf gobuster feroxbuster nmap masscan \
  hydra medusa ncrack sslscan tshark hping3 mitmproxy sslyze subfinder httpx \
  amass arjun mtr whatweb dirb socat netcat ngrep iperf3 whois dig wrk hey; do
  if command -v "$tool" > /dev/null 2>&1; then
    echo "OK: $tool"
  else
    echo "MISSING: $tool"
  fi
done
```

### Verificar o Executor de Suítes

Confirme que o executor e as suítes estão instalados:

```bash
ls /opt/traffic-generator/suites/runner.sh
ls /opt/traffic-generator/suites/*/

/opt/traffic-generator/suites/runner.sh web-app-attacks --dry-run
```

## Integração com Componentes Upstream

O gerador de tráfego tem como alvo um balanceador de carga HTTP F5 XC que está à frente do servidor de origem. Implante o servidor de origem primeiro, configure o F5 XC e então defina `target_fqdn` com o FQDN do balanceador de carga:

```bash
# Após implantar o origin-server e criar um balanceador de carga HTTP F5 XC:
cat > terraform.tfvars <<EOF
subscription_id  = "your-subscription-id"
target_fqdn      = "your-xc-load-balancer.example.com"
target_origin_ip = "$(cd ../../origin-server/terraform && terraform output -raw public_ip)"
EOF
```

| Variável Obrigatória | Origem | Descrição |
|---|---|---|
| `target_fqdn` | Console F5 XC | FQDN do balanceador de carga HTTP que protege o servidor de origem |
| `target_origin_ip` | Saída `public_ip` do servidor de origem | IP direto da origem para testes de bypass (opcional) |

Prossiga para o [Catálogo de Ferramentas](../04-tool-catalog/) para a referência completa de ferramentas ou [Suítes](../05-suites/) para começar a gerar tráfego.
