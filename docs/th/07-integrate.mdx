## สถาปัตยกรรมแบบเต็ม

ตัวสร้างทราฟฟิกเป็นหนึ่งในส่วนประกอบของสภาพแวดล้อมสาธิตแบบหลายชั้น สถาปัตยกรรมที่สมบูรณ์เมื่อมีการปรับใช้ส่วนประกอบทั้งหมด:

```
Traffic Generator -> F5 XC HTTP LB (WAF/Bot/API/CSD) -> Origin Server
                         |
               CDN Simulator (optional)
```

```mermaid
graph LR
    TG[Traffic Generator VM] -->|Attack Traffic<br/>HTTPS| XCHLB[F5 XC HTTP LB]
    XCHLB -->|WAF / Bot / API / CSD| ORIGIN[Origin Server VM<br/>Juice Shop, DVWA,<br/>VAmPI, httpbin, whoami]
    CDN[CDN Simulator<br/>optional] -->|Cached Requests| XCHLB
    TG -.->|Direct Baseline<br/>HTTP optional| ORIGIN
```

แต่ละส่วนประกอบจะถูกปรับใช้และกำหนดค่าอย่างอิสระผ่าน Terraform ตัวสร้างทราฟฟิกกำหนดเป้าหมายไปที่ FQDN ของ F5 XC load balancer ไม่ใช่ไปที่ origin server โดยตรง

### ลำดับการปรับใช้

1. ปรับใช้ **origin server** ก่อน -- มันจัดเตรียมแอปพลิเคชันแบ็กเอนด์
2. กำหนดค่า **F5 XC HTTP load balancer** โดยใช้ origin server เป็น origin pool
3. แนบ **นโยบาย WAF, Bot Defense, API Security และ CSD** เข้ากับ load balancer
4. ปรับใช้ **ตัวสร้างทราฟฟิก** โดยตั้งค่า `target_fqdn` เป็นโดเมน F5 XC LB

### การกำหนดค่าเป้าหมาย

`config.env` ของตัวสร้างทราฟฟิกเชื่อมต่อกับส่วนที่เหลือของสถาปัตยกรรม:

```bash
# Target the F5 XC load balancer (traffic passes through security policies)
TARGET_FQDN=demo.example.com

# Optional: target the origin server directly (bypasses F5 XC)
TARGET_ORIGIN_IP=20.10.5.100
```

เมื่อตั้งค่า `TARGET_FQDN` แล้ว สคริปต์ชุดทั้งหมดจะส่งทราฟฟิกไปที่ `https://<TARGET_FQDN>/...` F5 XC load balancer รับคำร้องขอ ใช้นโยบายความปลอดภัย และส่งต่อทราฟฟิกที่อนุญาตไปยัง origin server

## การผสานรวม CSD Demo

ชุด `javascript-exploits` ได้รับการออกแบบมาโดยเฉพาะสำหรับการสาธิต Client-Side Defense บน origin server ชุดนี้ตรวจสอบความถูกต้องของฟังก์ชันการทำงาน CSD Phase 2:

**ขั้นตอน Phase 2:**

1. Origin server โฮสต์หน้า CSD demo ที่ `/csd-demo/`
2. F5 XC CSD ฉีด JavaScript สำหรับการตรวจสอบเข้าไปในหน้า
3. ชุด javascript-exploits ของตัวสร้างทราฟฟิกพยายาม:
   - ฉีดสคริปต์แบบอินไลน์ที่เลียนแบบ Magecart skimmers
   - แก้ไของค์ประกอบ DOM เพื่อเปลี่ยนเส้นทางการส่งฟอร์ม
   - โหลด JavaScript ของบุคคลที่สามที่ไม่ได้รับอนุญาต
4. F5 XC CSD ตรวจจับการแก้ไขเหล่านี้และรายงานในแดชบอร์ด CSD

วิธีใช้ชุด javascript-exploits:

```bash
# Ensure CSD is enabled on the F5 XC HTTP LB for the /csd-demo/ path
# Then run the suite
/opt/traffic-generator/suites/runner.sh javascript-exploits
```

## การผสานรวม CDN Simulator

เมื่อมีการปรับใช้ CDN Simulator สถาปัตยกรรมจะเพิ่มชั้นแคช:

```
Traffic Generator -> CDN Simulator -> F5 XC HTTP LB -> Origin Server
```

CDN Simulator อยู่ด้านหน้าของ F5 XC load balancer ทำหน้าที่แคชการตอบกลับและเพิ่มเฮดเดอร์ที่คล้ายกับ CDN หากต้องการกำหนดเป้าหมายทราฟฟิกผ่าน CDN:

```bash
# Set TARGET_FQDN to the CDN Simulator's endpoint instead of F5 XC directly
TARGET_FQDN=cdn.demo.example.com
```

สิ่งนี้มีประโยชน์สำหรับการสาธิตวิธีที่ F5 XC จัดการทราฟฟิกที่มาผ่าน CDN รวมถึง:

- การระบุ IP จริงของไคลเอนต์เบื้องหลังเฮดเดอร์พร็อกซีของ CDN
- การใช้กฎ WAF กับคำร้องขอที่อาจถูกแก้ไขโดย CDN
- การจำแนกประเภท Bot Defense เมื่อ CDN แก้ไขลายนิ้วมือของเบราว์เซอร์

## การเปรียบเทียบทราฟฟิกแบบตรงกับผ่าน LB

ตัวสร้างทราฟฟิกรองรับการส่งทราฟฟิกทั้งผ่าน F5 XC และส่งตรงไปยัง origin การเปรียบเทียบนี้แสดงให้เห็นคุณค่าของฟีเจอร์ความปลอดภัยของ F5 XC:

### ผ่าน F5 XC (ค่าเริ่มต้น)

```bash
# Traffic goes: Generator -> F5 XC LB -> Origin
TARGET_FQDN=demo.example.com /opt/traffic-generator/suites/runner.sh web-app-attacks
```

ผลลัพธ์ที่คาดหวัง: WAF บล็อก SQL injection, XSS และ command injection payloads แดชบอร์ด Security Events แสดงคำร้องขอที่ถูกบล็อกพร้อมรายละเอียดการละเมิด

### ส่งตรงไปยัง Origin (เส้นฐาน)

```bash
# Traffic goes: Generator -> Origin (no security layer)
TARGET_FQDN=20.10.5.100 /opt/traffic-generator/suites/runner.sh web-app-attacks
```

ผลลัพธ์ที่คาดหวัง: payloads ทั้งหมดเข้าถึงแอปพลิเคชัน origin โดยไม่มีการกรอง Juice Shop และ DVWA ประมวลผล attack payloads นี่แสดงให้เห็นว่าจะเกิดอะไรขึ้นหากไม่มีการป้องกันจาก F5 XC

### ขั้นตอนการสาธิตแบบเคียงข้างกัน

สำหรับการสาธิตที่น่าสนใจ ให้รันชุดเดียวกันทั้งสองวิธี:

1. รัน `web-app-attacks` ส่งตรงไปยัง origin -- แสดงให้เห็นว่าการโจมตีสำเร็จ
2. รัน `web-app-attacks` ผ่าน F5 XC -- แสดงให้เห็นว่าการโจมตีถูกบล็อก
3. เปิดแดชบอร์ด F5 XC Security Events เพื่อแสดงคำร้องขอที่ถูกบล็อก
4. เปรียบเทียบผลลัพธ์ `meta.json` ของชุด: การรันแบบตรงแสดง "passed" มากกว่า (การโจมตีสำเร็จ) การรันผ่าน LB แสดง "failed" มากกว่า (การโจมตีถูกบล็อก)

```bash
TGEN_IP=$(terraform output -raw public_ip)
ORIGIN_IP="20.10.5.100"
LB_FQDN="demo.example.com"

# Run 1: Direct (baseline)
ssh azureuser@${TGEN_IP} "TARGET_FQDN=${ORIGIN_IP} /opt/traffic-generator/suites/runner.sh web-app-attacks"

# Run 2: Through F5 XC
ssh azureuser@${TGEN_IP} "TARGET_FQDN=${LB_FQDN} /opt/traffic-generator/suites/runner.sh web-app-attacks"

# Compare results
ssh azureuser@${TGEN_IP} 'for d in $(ls -t /opt/traffic-generator/results/ | head -2); do echo "=== $d ==="; cat /opt/traffic-generator/results/$d/meta.json; echo; done'
```

## การปรับใช้ Terraform แบบหลายส่วนประกอบ

เมื่อปรับใช้สภาพแวดล้อมแล็บเต็มรูปแบบ ให้ใช้ workspace หรือไดเรกทอรี Terraform แยกต่างหากสำหรับแต่ละส่วนประกอบ:

```bash
# 1. Deploy origin server
cd origin-server
terraform apply -var="subscription_id=YOUR_SUB_ID"
ORIGIN_IP=$(terraform output -raw public_ip)

# 2. Configure F5 XC (manual or via separate Terraform)
# Create origin pool -> HTTP LB -> attach WAF/Bot/API/CSD policies
# LB_FQDN=demo.example.com

# 3. Deploy traffic generator targeting the F5 XC LB
cd ../traffic-generator
terraform apply \
  -var="subscription_id=YOUR_SUB_ID" \
  -var="target_fqdn=demo.example.com" \
  -var="target_origin_ip=${ORIGIN_IP}"

# 4. Generate traffic
TGEN_IP=$(terraform output -raw public_ip)
ssh azureuser@${TGEN_IP} '/opt/traffic-generator/suites/runner.sh web-app-attacks'
```
