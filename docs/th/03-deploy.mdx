ไฟล์ Terraform ทั้งหมดอยู่ในไดเรกทอรี [`terraform/`](https://github.com/f5xc-salesdemos/traffic-generator/tree/main/terraform) โคลน repository และ deploy ได้โดยตรง:

```bash
git clone https://github.com/f5xc-salesdemos/traffic-generator.git
cd traffic-generator/terraform
cp terraform.tfvars.example terraform.tfvars
# Edit terraform.tfvars with your Azure subscription ID and target FQDN
```

## การตั้งค่า Terraform

### โครงสร้างไฟล์ Terraform

ไดเรกทอรี terraform ประกอบด้วย 9 ไฟล์ตาม Demo Resource Standard:

- `versions.tf` -- ข้อกำหนดเวอร์ชันของ Terraform และ provider (azurerm ~> 4.0, azuread ~> 3.0)
- `providers.tf` -- การตั้งค่า Azure RM และ Azure AD provider
- `data.tf` -- แหล่งข้อมูล Azure AD สำหรับการระบุตัวตนผู้ deploy อัตโนมัติ
- `locals.tf` -- การระบุตัวตนผู้ deploy, การตั้งชื่อทรัพยากรตาม Azure Cloud Adoption Framework, แท็กมาตรฐาน
- `main.tf` -- Resource group (ชื่อ `rg-traffic-generator-{environment}-{deployer}`)
- `variables.tf` -- ตัวแปร input ทั้งหมด (จำเป็น 2 ตัว, ไม่บังคับ 9 ตัว) พร้อมการระบุตัวตนผู้ deploy อัตโนมัติผ่าน Azure AD
- `network.tf` -- VNet (10.201.0.0/16), subnet, public IP, NSG (พอร์ต 22 SSH), NIC
- `vm.tf` -- Ubuntu 24.04 VM พร้อม cloud-init ผ่าน templatefile()
- `outputs.tf` -- เอาต์พุต 17 รายการ: เอาต์พุตมาตรฐาน 15 รายการที่ใช้ร่วมกันกับทรัพยากร demo ทั้งหมด (deployer, public_ip, private_ip, ssh_command, resource_group_name, vm_name, nsg_name, vnet_name, subnet_id, component, environment, resource_group_id, vm_id, nsg_id, location) บวกเอาต์พุตเฉพาะคอมโพเนนต์ 2 รายการ (target_fqdn, status_check)

### Provider และตัวแปร

`main.tf` ตั้งค่า Azure provider:

```hcl file=./_data/main.tf
```

`variables.tf` กำหนดพารามิเตอร์ที่สามารถปรับแต่งได้ทั้งหมด ค่าเริ่มต้นของ `vm_size` คือ `Standard_F16s_v2` (16 vCPU compute-optimized) -- ดู [การกำหนดขนาด VM](#การกำหนดขนาด-vm) สำหรับแนวทางการปรับขนาด:

```hcl file=./_data/variables.tf
```

### โครงสร้างพื้นฐานเครือข่าย

`network.tf` สร้าง VNet, subnet, public IP, NSG (พอร์ต 22 SSH) และ NIC:

```hcl file=./_data/network.tf
```

### Virtual Machine พร้อม Cloud-Init

`vm.tf` สร้าง Ubuntu 24.04 VM และส่งเทมเพลต cloud-init พร้อมตัวแปร target และ tooling:

```hcl file=./_data/vm.tf
```

### การจัดเตรียมด้วย Cloud-Init

`cloud-init.yaml` เป็นการตั้งค่าที่ผ่านการตรวจสอบสำหรับ production ภายใต้การทดสอบโหลดต่อเนื่องมากกว่า 48 ชั่วโมง รวมถึงการทดสอบ origin torture 4 รอบ ใช้ helper library พร้อม retry logic, การบันทึกความคืบหน้าไปยัง `/var/log/cloud-init-progress.log`, ขั้นตอนการจัดเตรียม 9 ขั้นตอน, และการรายงานสถานะแบบมีเงื่อนไข (ready/degraded ใน `status.json`) จัดเตรียมสิ่งต่อไปนี้:

- **การปรับแต่ง Kernel** -- `somaxconn=131072`, `tcp_max_tw_buckets=4000000`, `tcp_fin_timeout=5`, socket buffers ขนาด 64 MiB, `file-max=4194304`
- **การปรับแต่ง NIC** -- RPS/RFS ข้ามทุกคอร์, ring buffers สูงสุดผ่าน ethtool, THP=always
- **แพ็คเกจ APT** -- nikto, nmap, masscan, sqlmap, hydra, medusa, ncrack, tshark, hping3, tcpdump, netcat, ngrep, iperf3, mtr, sslscan, socat, dirb, whatweb, wrk, hey, vegeta, ethtool
- **GitHub Releases** -- nuclei, dalfox, ffuf, gobuster, feroxbuster, subfinder, httpx, amass
- **แพ็คเกจ Python** -- mitmproxy, sslyze, dnsrecon, fierce, theHarvester, playwright, scapy, impacket, arjun
- **Node.js 20.x** -- Puppeteer พร้อม stealth plugin สำหรับการจำลอง bot
- **ระบบอัตโนมัติเบราว์เซอร์** -- Playwright Chromium สำหรับการทดสอบเบราว์เซอร์แบบ headless
- **Git clones** -- testssl.sh, recon-ng, spiderfoot, SecLists, waf-bypass, PayloadsAllTheThings
- **การ deploy ชุดทดสอบ** -- ชุดทดสอบ traffic 19 ชุดที่โคลนจาก repository นี้ไปยัง `/opt/traffic-generator/suites/`
- **ไฟล์การตั้งค่า** -- `config.env` พร้อม `TARGET_FQDN`, `TARGET_ORIGIN_IP` และ `CRAPI_PORT`

cloud-init ใช้ตัวแปรเทมเพลต Terraform: `${target_fqdn}`, `${target_origin_ip}` และ `${tool_tier}` ตัวแปร Shell เช่น `$NPROC` จะถูก escape เป็น `$$NPROC` ใน Terraform templatefile

```yaml file=./_data/cloud-init.yaml
```

### เอาต์พุต

`outputs.tf` เปิดเผยเอาต์พุต 17 รายการ รวมถึง public/private IP, คำสั่ง SSH, ตัวระบุทรัพยากร และค่าเฉพาะคอมโพเนนต์ (target_fqdn, status_check):

```hcl file=./_data/outputs.tf
```

### ตัวอย่างไฟล์ตัวแปร

คัดลอก `terraform.tfvars.example` เป็น `terraform.tfvars` และกรอกค่าของคุณ `.gitignore` จะไม่รวม `terraform.tfvars` เพื่อป้องกันการ commit ข้อมูลประจำตัว:

```hcl file=./_data/terraform.tfvars.example
```

แทนที่ `subscription_id` ด้วย Azure subscription ID ของคุณ (จาก `az account show --query id -o tsv`) และ `target_fqdn` ด้วยโดเมน F5 XC load balancer ของคุณ

## Deploy

```bash
terraform init

terraform plan

terraform apply
```

Terraform จะแสดง public IP, คำสั่ง SSH และ target FQDN หลังจาก deployment สำเร็จ

## การกำหนดขนาด VM

แนะนำให้ใช้ VM ซีรีส์ F ที่ปรับแต่งสำหรับ compute สำหรับปริมาณงานการสร้าง traffic ที่ใช้ CPU เป็นหลักนี้ การปรับแต่ง kernel ของ cloud-init และ concurrency ของเครื่องมือจะปรับขนาดอัตโนมัติตามจำนวน vCPU

| VM SKU | vCPU | RAM | เครือข่าย | กรณีการใช้งาน |
|---|---|---|---|---|
| Standard_F4s_v2 | 4 | 8 GiB | 10 Gbps | Lab/demo เบาๆ |
| Standard_F8s_v2 | 8 | 16 GiB | 12.5 Gbps | Demo มาตรฐาน |
| Standard_F16s_v2 | 16 | 32 GiB | 12.5 Gbps | การทดสอบโหลด (ค่าเริ่มต้น) |
| Standard_F32s_v2 | 32 | 64 GiB | 16 Gbps | Benchmarking หนัก |
| Standard_D16s_v3 | 16 | 64 GiB | 12.5 Gbps | ปริมาณงานที่ใช้ RAM มาก |

:::tip
F16s_v2 ได้รับการตรวจสอบว่าเป็นตัวเลือกราคา-ประสิทธิภาพที่ดีที่สุดจากการทดสอบ A/B/C/D benchmark เทียบกับเป้าหมาย CDN F16s_v2 และ F32s_v2 ดู [benchmarks/](https://github.com/f5xc-salesdemos/traffic-generator/tree/main/benchmarks) สำหรับผลลัพธ์ทั้งหมด
:::

## การตรวจสอบหลัง Deploy

การจัดเตรียมด้วย Cloud-init ใช้เวลา **15-20 นาทีสำหรับ standard tier** และ **ประมาณ 25 นาทีสำหรับ full tier** สามารถเข้าถึง VM ผ่าน SSH ได้ทันที แต่เครื่องมือยังคงถูกติดตั้งอยู่จนกว่า cloud-init จะเสร็จสมบูรณ์

### รอให้ Cloud-Init เสร็จ

SSH เข้า VM และรอให้การจัดเตรียมเสร็จสิ้น:

```bash
ssh azureuser@$(terraform output -raw public_ip)

cloud-init status --wait
```

ผลลัพธ์ที่คาดหวังเมื่อเสร็จสมบูรณ์:

```
status: done
```

### ตรวจสอบไฟล์สถานะ

สคริปต์ cloud-init จะเขียนไฟล์สถานะเมื่อเสร็จสมบูรณ์:

```bash
cat /opt/traffic-generator/status.json
```

ผลลัพธ์ที่คาดหวัง:

```json
{
  "status": "ready",
  "tier": "standard",
  "target_fqdn": "demo.example.com",
  "completed": "2026-04-25T14:30:00Z",
  "tools_installed": true
}
```

### ตรวจสอบเครื่องมือสำคัญ

รัน verification loop เพื่อยืนยันว่าเครื่องมือสำคัญแต่ละตัวพร้อมใช้งานบน PATH:

```bash
for tool in nikto sqlmap nuclei dalfox ffuf gobuster feroxbuster nmap masscan \
  hydra medusa ncrack sslscan tshark hping3 mitmproxy sslyze subfinder httpx \
  amass arjun mtr whatweb dirb socat netcat ngrep iperf3 whois dig wrk hey; do
  if command -v "$tool" > /dev/null 2>&1; then
    echo "OK: $tool"
  else
    echo "MISSING: $tool"
  fi
done
```

### ตรวจสอบ Suite Runner

ยืนยันว่า runner และชุดทดสอบถูกติดตั้งแล้ว:

```bash
ls /opt/traffic-generator/suites/runner.sh
ls /opt/traffic-generator/suites/*/

/opt/traffic-generator/suites/runner.sh web-app-attacks --dry-run
```

## การเชื่อมต่อจากคอมโพเนนต์ต้นทาง

Traffic generator มีเป้าหมายเป็น F5 XC HTTP load balancer ที่อยู่หน้า origin server ให้ deploy origin server ก่อน ตั้งค่า F5 XC จากนั้นตั้งค่า `target_fqdn` เป็น FQDN ของ load balancer:

```bash
# After deploying origin-server and creating an F5 XC HTTP load balancer:
cat > terraform.tfvars <<EOF
subscription_id  = "your-subscription-id"
target_fqdn      = "your-xc-load-balancer.example.com"
target_origin_ip = "$(cd ../../origin-server/terraform && terraform output -raw public_ip)"
EOF
```

| ตัวแปรที่จำเป็น | แหล่งที่มา | คำอธิบาย |
|---|---|---|
| `target_fqdn` | F5 XC console | FQDN ของ HTTP load balancer ที่ปกป้อง origin server |
| `target_origin_ip` | เอาต์พุต `public_ip` ของ Origin server | IP ตรงของ origin สำหรับการทดสอบ bypass (ไม่บังคับ) |

ดำเนินการต่อที่ [แคตตาล็อกเครื่องมือ](../04-tool-catalog/) สำหรับข้อมูลอ้างอิงเครื่องมือทั้งหมด หรือ [ชุดทดสอบ](../05-suites/) เพื่อเริ่มสร้าง traffic
