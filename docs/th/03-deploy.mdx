ไฟล์ Terraform ทั้งหมดอยู่ในไดเรกทอรี [`terraform/`](https://github.com/f5xc-salesdemos/traffic-generator/tree/main/terraform) โคลนรีพอซิทอรีและทำการติดตั้งโดยตรง:

```bash
git clone https://github.com/f5xc-salesdemos/traffic-generator.git
cd traffic-generator/terraform
cp terraform.tfvars.example terraform.tfvars
# Edit terraform.tfvars with your Azure subscription ID and target FQDN
```

## การกำหนดค่า Terraform

### โครงสร้างไฟล์ Terraform

ไดเรกทอรี terraform ประกอบด้วย 9 ไฟล์ตามมาตรฐาน Demo Resource Standard:

- `versions.tf` -- ข้อจำกัดเวอร์ชันของ Terraform และ provider (azurerm ~> 4.0, azuread ~> 3.0)
- `providers.tf` -- การกำหนดค่า Azure RM และ Azure AD provider
- `data.tf` -- แหล่งข้อมูล Azure AD สำหรับการตรวจหาผู้ติดตั้งอัตโนมัติ
- `locals.tf` -- การตรวจหาผู้ติดตั้ง, การตั้งชื่อทรัพยากรตาม Azure Cloud Adoption Framework, แท็กมาตรฐาน
- `main.tf` -- กลุ่มทรัพยากร (ตั้งชื่อเป็น `rg-traffic-generator-{environment}-{deployer}`)
- `variables.tf` -- ตัวแปรอินพุตทั้งหมด (จำเป็น 2 ตัว, เลือกได้ 9 ตัว) พร้อมการตรวจหาผู้ติดตั้งอัตโนมัติผ่าน Azure AD
- `network.tf` -- VNet (10.201.0.0/16), ซับเน็ต, public IP, NSG (พอร์ต 22 SSH), NIC
- `vm.tf` -- VM Ubuntu 24.04 พร้อม cloud-init ผ่าน templatefile()
- `outputs.tf` -- เอาต์พุต 17 รายการ: 15 เอาต์พุตมาตรฐานที่ใช้ร่วมกันโดยทรัพยากรสาธิตทั้งหมด (deployer, public_ip, private_ip, ssh_command, resource_group_name, vm_name, nsg_name, vnet_name, subnet_id, component, environment, resource_group_id, vm_id, nsg_id, location) บวกกับ 2 เอาต์พุตเฉพาะคอมโพเนนต์ (target_fqdn, status_check)

### Provider และตัวแปร

`main.tf` กำหนดค่า Azure provider:

```hcl file=./_data/main.tf
```

`variables.tf` กำหนดพารามิเตอร์ที่ปรับแต่งได้ทั้งหมด ค่าเริ่มต้นของ `vm_size` คือ `Standard_F16s_v2` (16 vCPU ปรับแต่งสำหรับการประมวลผล) -- ดู [การเลือกขนาด VM](#การเลือกขนาด-vm) สำหรับคำแนะนำในการปรับขนาด:

```hcl file=./_data/variables.tf
```

### โครงสร้างพื้นฐานเครือข่าย

`network.tf` สร้าง VNet, ซับเน็ต, public IP, NSG (พอร์ต 22 SSH), และ NIC:

```hcl file=./_data/network.tf
```

### เครื่องเสมือนพร้อม Cloud-Init

`vm.tf` สร้าง VM Ubuntu 24.04 และส่งต่อเทมเพลต cloud-init พร้อมตัวแปรเป้าหมายและเครื่องมือ:

```hcl file=./_data/vm.tf
```

### การจัดเตรียม Cloud-Init

`cloud-init.yaml` เป็นการกำหนดค่าที่ผ่านการตรวจสอบแล้วในสภาพแวดล้อมจริง โดยได้รับการยืนยันภายใต้การทดสอบโหลดต่อเนื่องกว่า 48 ชั่วโมง รวมถึงการทดสอบ origin torture 4 รอบ ใช้ไลบรารีตัวช่วยพร้อมตรรกะการลองใหม่, การบันทึกความคืบหน้าไปยัง `/var/log/cloud-init-progress.log`, 9 ขั้นตอนการจัดเตรียม, และการรายงานสถานะแบบมีเงื่อนไข (ready/degraded ใน `status.json`) โดยจัดเตรียม:

- **การปรับแต่ง Kernel** -- `somaxconn=131072`, `tcp_max_tw_buckets=4000000`, `tcp_fin_timeout=5`, บัฟเฟอร์ซ็อกเก็ต 64 MiB, `file-max=4194304`
- **การปรับแต่ง NIC** -- RPS/RFS ข้ามทุกคอร์, เพิ่มบัฟเฟอร์ริงสูงสุดผ่าน ethtool, THP=always
- **แพ็กเกจ APT** -- nikto, nmap, masscan, sqlmap, hydra, medusa, ncrack, tshark, hping3, tcpdump, netcat, ngrep, iperf3, mtr, sslscan, socat, dirb, whatweb, wrk, hey, vegeta, ethtool
- **GitHub Releases** -- nuclei, dalfox, ffuf, gobuster, feroxbuster, subfinder, httpx, amass
- **แพ็กเกจ Python** -- mitmproxy, sslyze, dnsrecon, fierce, theHarvester, playwright, scapy, impacket, arjun
- **Node.js 20.x** -- Puppeteer พร้อมปลั๊กอิน stealth สำหรับการจำลองบอท
- **การทำงานอัตโนมัติของเบราว์เซอร์** -- Playwright Chromium สำหรับการทดสอบเบราว์เซอร์แบบ headless
- **Git clones** -- testssl.sh, recon-ng, spiderfoot, SecLists, waf-bypass, PayloadsAllTheThings
- **การติดตั้งชุด Suite** -- 19 ชุดทราฟฟิกโคลนจากรีพอซิทอรีนี้ไปยัง `/opt/traffic-generator/suites/`
- **ไฟล์การกำหนดค่า** -- `config.env` พร้อม `TARGET_FQDN`, `TARGET_ORIGIN_IP`, และ `CRAPI_PORT`

Cloud-init ใช้ตัวแปรเทมเพลต Terraform: `${target_fqdn}`, `${target_origin_ip}`, และ `${tool_tier}` ตัวแปร Shell เช่น `$NPROC` จะถูก escape เป็น `$$NPROC` ใน Terraform templatefile

```yaml file=./_data/cloud-init.yaml
```

### เอาต์พุต

`outputs.tf` แสดงเอาต์พุต 17 รายการ รวมถึง public/private IP, คำสั่ง SSH, ตัวระบุทรัพยากร, และค่าเฉพาะคอมโพเนนต์ (target_fqdn, status_check):

```hcl file=./_data/outputs.tf
```

### ตัวอย่างไฟล์ตัวแปร

คัดลอก `terraform.tfvars.example` ไปเป็น `terraform.tfvars` และกรอกค่าของคุณ `.gitignore` จะยกเว้น `terraform.tfvars` เพื่อป้องกันการ commit ข้อมูลประจำตัว:

```hcl file=./_data/terraform.tfvars.example
```

แทนที่ `subscription_id` ด้วย Azure subscription ID ของคุณ (จาก `az account show --query id -o tsv`) และ `target_fqdn` ด้วยโดเมน F5 XC load balancer ของคุณ

## การติดตั้ง

```bash
terraform init

terraform plan

terraform apply
```

Terraform จะแสดง public IP, คำสั่ง SSH, และ target FQDN หลังจากการติดตั้งสำเร็จ

## การเลือกขนาด VM

VM ชนิด F-series ที่ปรับแต่งสำหรับการประมวลผลเป็นตัวเลือกที่แนะนำสำหรับภาระงานการสร้างทราฟฟิกที่ใช้ CPU เป็นหลัก การปรับแต่ง kernel ของ cloud-init และความพร้อมกันของเครื่องมือจะปรับขนาดอัตโนมัติตามจำนวน vCPU

| SKU ของ VM | vCPU | RAM | เครือข่าย | กรณีการใช้งาน |
|---|---|---|---|---|
| Standard_F4s_v2 | 4 | 8 GiB | 10 Gbps | แล็บ/สาธิตขนาดเบา |
| Standard_F8s_v2 | 8 | 16 GiB | 12.5 Gbps | การสาธิตมาตรฐาน |
| Standard_F16s_v2 | 16 | 32 GiB | 12.5 Gbps | การทดสอบโหลด (ค่าเริ่มต้น) |
| Standard_F32s_v2 | 32 | 64 GiB | 16 Gbps | การเปรียบเทียบประสิทธิภาพขนาดหนัก |
| Standard_D16s_v3 | 16 | 64 GiB | 12.5 Gbps | ภาระงานที่ใช้ RAM สูง |

:::tip
F16s_v2 ได้รับการยืนยันว่าเป็นตัวเลือกที่คุ้มค่าที่สุดจากการทดสอบเปรียบเทียบ A/B/C/D กับเป้าหมาย CDN F16s_v2 และ F32s_v2 ดู [benchmarks/](https://github.com/f5xc-salesdemos/traffic-generator/tree/main/benchmarks) สำหรับผลลัพธ์ทั้งหมด
:::

## การตรวจสอบหลังการติดตั้ง

การจัดเตรียม Cloud-init ใช้เวลา **15-20 นาทีสำหรับระดับมาตรฐาน** และ **ประมาณ 25 นาทีสำหรับระดับเต็ม** VM สามารถเข้าถึงได้ผ่าน SSH ทันที แต่เครื่องมือยังคงกำลังติดตั้งจนกว่า cloud-init จะเสร็จสมบูรณ์

### รอให้ Cloud-Init เสร็จสิ้น

SSH เข้าสู่ VM และรอให้การจัดเตรียมเสร็จสิ้น:

```bash
ssh azureuser@$(terraform output -raw public_ip)

cloud-init status --wait
```

เอาต์พุตที่คาดหวังเมื่อเสร็จสมบูรณ์:

```
status: done
```

### ตรวจสอบไฟล์สถานะ

สคริปต์ cloud-init จะเขียนไฟล์สถานะเมื่อเสร็จสมบูรณ์:

```bash
cat /opt/traffic-generator/status.json
```

เอาต์พุตที่คาดหวัง:

```json
{
  "status": "ready",
  "tier": "standard",
  "target_fqdn": "demo.example.com",
  "completed": "2026-04-25T14:30:00Z",
  "tools_installed": true
}
```

### ตรวจสอบเครื่องมือสำคัญ

เรียกใช้ลูปตรวจสอบเพื่อยืนยันว่าเครื่องมือสำคัญแต่ละตัวพร้อมใช้งานบน PATH:

```bash
for tool in nikto sqlmap nuclei dalfox ffuf gobuster feroxbuster nmap masscan \
  hydra medusa ncrack sslscan tshark hping3 mitmproxy sslyze subfinder httpx \
  amass arjun mtr whatweb dirb socat netcat ngrep iperf3 whois dig wrk hey; do
  if command -v "$tool" > /dev/null 2>&1; then
    echo "OK: $tool"
  else
    echo "MISSING: $tool"
  fi
done
```

### ตรวจสอบ Suite Runner

ยืนยันว่า runner และชุด suite ได้รับการติดตั้งแล้ว:

```bash
ls /opt/traffic-generator/suites/runner.sh
ls /opt/traffic-generator/suites/*/

/opt/traffic-generator/suites/runner.sh web-app-attacks --dry-run
```

## การเชื่อมต่อจากคอมโพเนนต์ต้นทาง

Traffic generator มุ่งเป้าไปที่ F5 XC HTTP load balancer ที่อยู่หน้า origin server ติดตั้ง origin server ก่อน กำหนดค่า F5 XC จากนั้นตั้ง `target_fqdn` เป็น FQDN ของ load balancer:

```bash
# After deploying origin-server and creating an F5 XC HTTP load balancer:
cat > terraform.tfvars <<EOF
subscription_id  = "your-subscription-id"
target_fqdn      = "your-xc-load-balancer.example.com"
target_origin_ip = "$(cd ../../origin-server/terraform && terraform output -raw public_ip)"
EOF
```

| ตัวแปรที่จำเป็น | แหล่งที่มา | คำอธิบาย |
|---|---|---|
| `target_fqdn` | คอนโซล F5 XC | FQDN ของ HTTP load balancer ที่ป้องกัน origin server |
| `target_origin_ip` | เอาต์พุต `public_ip` ของ origin server | IP ตรงของ origin สำหรับการทดสอบ bypass (ไม่บังคับ) |

ดำเนินการต่อที่ [แคตตาล็อกเครื่องมือ](../04-tool-catalog/) สำหรับข้อมูลอ้างอิงเครื่องมือทั้งหมด หรือ [ชุด Suite](../05-suites/) เพื่อเริ่มสร้างทราฟฟิก
