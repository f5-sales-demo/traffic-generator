ไฟล์ Terraform ทั้งหมดอยู่ในไดเรกทอรี [`terraform/`](https://github.com/f5xc-salesdemos/traffic-generator/tree/main/terraform) โคลน repository และปรับใช้งานโดยตรง:

```bash
git clone https://github.com/f5xc-salesdemos/traffic-generator.git
cd traffic-generator/terraform
cp terraform.tfvars.example terraform.tfvars
# Edit terraform.tfvars with your Azure subscription ID and target FQDN
```

## การกำหนดค่า Terraform

### โครงสร้างไฟล์ Terraform

ไดเรกทอรี terraform ประกอบด้วย 9 ไฟล์ตามมาตรฐาน Demo Resource Standard:

- `versions.tf` -- ข้อจำกัดเวอร์ชัน Terraform และ provider (azurerm ~> 4.0, azuread ~> 3.0)
- `providers.tf` -- การกำหนดค่า provider สำหรับ Azure RM และ Azure AD
- `data.tf` -- แหล่งข้อมูล Azure AD สำหรับการแก้ไข deployer อัตโนมัติ
- `locals.tf` -- การแก้ไข deployer, การตั้งชื่อทรัพยากรตาม Azure Cloud Adoption Framework, แท็กมาตรฐาน
- `main.tf` -- Resource group (ตั้งชื่อเป็น `rg-traffic-generator-{environment}-{deployer}`)
- `variables.tf` -- ตัวแปรนำเข้าทั้งหมด (2 ที่จำเป็น, 9 ที่เป็นตัวเลือก) พร้อมการแก้ไข deployer อัตโนมัติผ่าน Azure AD
- `network.tf` -- VNet (10.201.0.0/16), subnet, public IP, NSG (พอร์ต 22 SSH), NIC
- `vm.tf` -- Ubuntu 24.04 VM พร้อม cloud-init ผ่าน templatefile()
- `outputs.tf` -- 17 outputs: 15 outputs มาตรฐานที่ใช้ร่วมกันโดยทรัพยากรสาธิตทั้งหมด (deployer, public_ip, private_ip, ssh_command, resource_group_name, vm_name, nsg_name, vnet_name, subnet_id, component, environment, resource_group_id, vm_id, nsg_id, location) บวก 2 outputs เฉพาะส่วนประกอบ (target_fqdn, status_check)

### Provider และตัวแปร

`main.tf` กำหนดค่า Azure provider:


`variables.tf` กำหนดพารามิเตอร์ที่กำหนดค่าได้ทั้งหมด ค่าเริ่มต้นของ `vm_size` คือ `Standard_F16s_v2` (16 vCPU compute-optimized) -- ดู [การกำหนดขนาด VM](#vm-sizing) สำหรับคำแนะนำการปรับขนาด:

```hcl file=../_data/variables.tf
```

### การจัดเตรียมด้วย Cloud-Init

`cloud-init.yaml` คือการกำหนดค่าที่ผ่านการตรวจสอบในการใช้งานจริง ซึ่งได้รับการยืนยันภายใต้การทดสอบโหลดอย่างต่อเนื่องมากกว่า 48 ชั่วโมง รวมถึงการทดสอบ origin torture 4 รอบ ใช้ไลบรารีช่วยเหลือพร้อมตรรกะการลองใหม่, การบันทึกความคืบหน้าไปยัง `/var/log/cloud-init-progress.log`, 9 ขั้นตอนการจัดเตรียม และการรายงานสถานะแบบมีเงื่อนไข (ready/degraded ใน `status.json`) โดยจัดเตรียม:

- **การปรับแต่ง Kernel** -- `somaxconn=131072`, `tcp_max_tw_buckets=4000000`, `tcp_fin_timeout=5`, socket buffers ขนาด 64 MiB, `file-max=4194304`
- **การปรับแต่ง NIC** -- RPS/RFS ทั่วทุก core, ring buffers สูงสุดผ่าน ethtool, THP=always
- **แพ็กเกจ APT** -- nikto, nmap, masscan, sqlmap, hydra, medusa, ncrack, tshark, hping3, tcpdump, netcat, ngrep, iperf3, mtr, sslscan, socat, dirb, whatweb, wrk, hey, vegeta, ethtool
- **GitHub Releases** -- nuclei, dalfox, ffuf, gobuster, feroxbuster, subfinder, httpx, amass
- **แพ็กเกจ Python** -- mitmproxy, sslyze, dnsrecon, fierce, theHarvester, playwright, scapy, impacket, arjun
- **Node.js 20.x** -- Puppeteer พร้อม stealth plugin สำหรับการจำลอง bot
- **การทำงานอัตโนมัติของเบราว์เซอร์** -- Playwright Chromium สำหรับการทดสอบ headless browser
- **Git clones** -- testssl.sh, recon-ng, spiderfoot, SecLists, waf-bypass, PayloadsAllTheThings
- **การปรับใช้งาน Suite** -- ชุดทราฟฟิก 19 ชุด โคลนจาก repository นี้ไปยัง `/opt/traffic-generator/suites/`
- **ไฟล์กำหนดค่า** -- `config.env` พร้อม `TARGET_FQDN`, `TARGET_ORIGIN_IP` และ `CRAPI_PORT`

cloud-init ใช้ตัวแปร Terraform template: `${target_fqdn}`, `${target_origin_ip}` และ `${tool_tier}` ตัวแปร Shell เช่น `$NPROC` ถูก escape เป็น `$$NPROC` ใน Terraform templatefile

```yaml file=../_data/cloud-init.yaml
```

## ปรับใช้งาน

```bash
terraform init

terraform plan

terraform apply
```

Terraform แสดงผล public IP, คำสั่ง SSH และ target FQDN หลังจากการปรับใช้งานสำเร็จ

## การกำหนดขนาด VM

แนะนำให้ใช้ F-series compute-optimized VMs สำหรับ workload การสร้างทราฟฟิกที่ใช้ CPU เป็นหลัก การปรับแต่ง kernel ของ cloud-init และการทำงานพร้อมกันของเครื่องมือจะปรับขนาดโดยอัตโนมัติตามจำนวน vCPU

| VM SKU | vCPU | RAM | เครือข่าย | กรณีการใช้งาน |
|---|---|---|---|---|
| Standard_F4s_v2 | 4 | 8 GiB | 10 Gbps | Lab/demo ขนาดเล็ก |
| Standard_F8s_v2 | 8 | 16 GiB | 12.5 Gbps | Demo มาตรฐาน |
| Standard_F16s_v2 | 16 | 32 GiB | 12.5 Gbps | การทดสอบโหลด (ค่าเริ่มต้น) |
| Standard_F32s_v2 | 32 | 64 GiB | 16 Gbps | การทดสอบประสิทธิภาพสูง |
| Standard_D16s_v3 | 16 | 64 GiB | 12.5 Gbps | Workload ที่ต้องการ RAM มาก |

:::tip
F16s_v2 ได้รับการตรวจสอบว่าเป็นตัวเลือกที่มีอัตราส่วนราคาต่อประสิทธิภาพดีที่สุดจากการทดสอบ benchmark A/B/C/D กับเป้าหมาย CDN F16s_v2 และ F32s_v2 ดูผลลัพธ์เต็มรูปแบบได้ที่ [benchmarks/](https://github.com/f5xc-salesdemos/traffic-generator/tree/main/benchmarks)
:::

## การตรวจสอบหลังปรับใช้งาน

การจัดเตรียมด้วย cloud-init ใช้เวลา **15-20 นาทีสำหรับ standard tier** และ **ประมาณ 25 นาทีสำหรับ full tier** VM สามารถเข้าถึงผ่าน SSH ได้ทันที แต่เครื่องมือยังคงถูกติดตั้งอยู่จนกว่า cloud-init จะเสร็จสมบูรณ์

### รอ Cloud-Init

SSH เข้าสู่ VM และรอให้การจัดเตรียมเสร็จสมบูรณ์:

```bash
ssh azureuser@$(terraform output -raw public_ip)

cloud-init status --wait
```

ผลลัพธ์ที่คาดหวังเมื่อเสร็จสมบูรณ์:

```
status: done
```

### ตรวจสอบไฟล์สถานะ

สคริปต์ cloud-init เขียนไฟล์สถานะเมื่อเสร็จสมบูรณ์:

```bash
cat /opt/traffic-generator/status.json
```

ผลลัพธ์ที่คาดหวัง:

```json
{
  "status": "ready",
  "tier": "standard",
  "target_fqdn": "demo.example.com",
  "completed": "2026-04-25T14:30:00Z",
  "tools_installed": true
}
```

### ตรวจสอบเครื่องมือสำคัญ

รัน verification loop เพื่อยืนยันว่าเครื่องมือสำคัญแต่ละรายการพร้อมใช้งานบน PATH:

```bash
for tool in nikto sqlmap nuclei dalfox ffuf gobuster feroxbuster nmap masscan \
  hydra medusa ncrack sslscan tshark hping3 mitmproxy sslyze subfinder httpx \
  amass arjun mtr whatweb dirb socat netcat ngrep iperf3 whois dig wrk hey; do
  if command -v "$tool" > /dev/null 2>&1; then
    echo "OK: $tool"
  else
    echo "MISSING: $tool"
  fi
done
```

### ตรวจสอบ Suite Runner

ยืนยันว่า runner และ suites ถูกติดตั้งแล้ว:

```bash
ls /opt/traffic-generator/suites/runner.sh
ls /opt/traffic-generator/suites/*/

/opt/traffic-generator/suites/runner.sh web-app-attacks --dry-run
```

## การเชื่อมต่อจากส่วนประกอบต้นน้ำ

ตัวสร้างทราฟฟิกกำหนดเป้าหมายที่ F5 XC HTTP load balancer ที่อยู่หน้าเซิร์ฟเวอร์ต้นทาง ปรับใช้งานเซิร์ฟเวอร์ต้นทางก่อน จากนั้นกำหนดค่า F5 XC แล้วตั้งค่า `target_fqdn` เป็น FQDN ของ load balancer:

```bash
# After deploying origin-server and creating an F5 XC HTTP load balancer:
cat > terraform.tfvars <<EOF
subscription_id  = "your-subscription-id"
target_fqdn      = "your-xc-load-balancer.example.com"
target_origin_ip = "$(cd ../../origin-server/terraform && terraform output -raw public_ip)"
EOF
```

| ตัวแปรที่จำเป็น | แหล่งที่มา | คำอธิบาย |
|---|---|---|
| `target_fqdn` | F5 XC คอนโซล | FQDN ของ HTTP load balancer ที่ปกป้องเซิร์ฟเวอร์ต้นทาง |
| `target_origin_ip` | output `public_ip` ของเซิร์ฟเวอร์ต้นทาง | IP ต้นทางโดยตรงสำหรับการทดสอบ bypass (ตัวเลือก) |

ดำเนินการต่อไปยัง [Tool Catalog](../04-tool-catalog/) สำหรับข้อมูลอ้างอิงเครื่องมือฉบับสมบูรณ์ หรือ [Suites](../05-suites/) เพื่อเริ่มสร้างทราฟฟิก
