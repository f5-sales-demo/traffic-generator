Alle Terraform-Dateien befinden sich im Verzeichnis [`terraform/`](https://github.com/f5xc-salesdemos/traffic-generator/tree/main/terraform). Klonen Sie das Repository und führen Sie die Bereitstellung direkt durch:

```bash
git clone https://github.com/f5xc-salesdemos/traffic-generator.git
cd traffic-generator/terraform
cp terraform.tfvars.example terraform.tfvars
# Bearbeiten Sie terraform.tfvars mit Ihrer Azure-Abonnement-ID und dem Ziel-FQDN
```

## Terraform-Konfiguration

### Terraform-Dateistruktur

Das Terraform-Verzeichnis enthält 9 Dateien gemäß dem Demo-Ressourcen-Standard:

- `versions.tf` -- Terraform- und Provider-Versionseinschränkungen (azurerm ~> 4.0, azuread ~> 3.0)
- `providers.tf` -- Azure RM- und Azure AD-Provider-Konfiguration
- `data.tf` -- Azure AD-Datenquellen für automatische Deployer-Auflösung
- `locals.tf` -- Deployer-Auflösung, Azure Cloud Adoption Framework-Ressourcenbenennung, Standard-Tags
- `main.tf` -- Ressourcengruppe (benannt `rg-traffic-generator-{environment}-{deployer}`)
- `variables.tf` -- Alle Eingabevariablen (2 erforderlich, 9 optional) mit automatischer Deployer-Auflösung über Azure AD
- `network.tf` -- VNet (10.201.0.0/16), Subnetz, öffentliche IP, NSG (Port 22 SSH), NIC
- `vm.tf` -- Ubuntu 24.04 VM mit cloud-init über templatefile()
- `outputs.tf` -- 17 Ausgaben: 15 Standardausgaben, die von allen Demo-Ressourcen gemeinsam genutzt werden (deployer, public_ip, private_ip, ssh_command, resource_group_name, vm_name, nsg_name, vnet_name, subnet_id, component, environment, resource_group_id, vm_id, nsg_id, location) plus 2 komponentenspezifische Ausgaben (target_fqdn, status_check)

### Provider und Variablen

`main.tf` konfiguriert den Azure-Provider:

```hcl file=./_data/main.tf
```

`variables.tf` definiert alle konfigurierbaren Parameter. Der Standardwert für `vm_size` ist `Standard_F16s_v2` (16 vCPU rechenoptimiert) -- siehe [VM-Dimensionierung](#vm-dimensionierung) für Skalierungshinweise:

```hcl file=./_data/variables.tf
```

### Netzwerkinfrastruktur

`network.tf` erstellt das VNet, Subnetz, die öffentliche IP, NSG (Port 22 SSH) und NIC:

```hcl file=./_data/network.tf
```

### Virtuelle Maschine mit Cloud-Init

`vm.tf` erstellt die Ubuntu 24.04 VM und übergibt das Cloud-Init-Template mit Ziel- und Tool-Variablen:

```hcl file=./_data/vm.tf
```

### Cloud-Init-Bereitstellung

`cloud-init.yaml` ist die produktionsvalidierte Konfiguration, die unter mehr als 48 Stunden kontinuierlichem Lasttesting einschließlich 4 Runden Origin-Torture-Testing verifiziert wurde. Sie verwendet eine Hilfsbibliothek mit Retry-Logik, Fortschrittsprotokollierung nach `/var/log/cloud-init-progress.log`, 9 Bereitstellungsphasen und bedingter Statusberichterstattung (ready/degraded in `status.json`). Sie stellt bereit:

- **Kernel-Tuning** -- `somaxconn=131072`, `tcp_max_tw_buckets=4000000`, `tcp_fin_timeout=5`, 64 MiB Socket-Puffer, `file-max=4194304`
- **NIC-Optimierung** -- RPS/RFS über alle Kerne, maximierte Ringpuffer über ethtool, THP=always
- **APT-Pakete** -- nikto, nmap, masscan, sqlmap, hydra, medusa, ncrack, tshark, hping3, tcpdump, netcat, ngrep, iperf3, mtr, sslscan, socat, dirb, whatweb, wrk, hey, vegeta, ethtool
- **GitHub Releases** -- nuclei, dalfox, ffuf, gobuster, feroxbuster, subfinder, httpx, amass
- **Python-Pakete** -- mitmproxy, sslyze, dnsrecon, fierce, theHarvester, playwright, scapy, impacket, arjun
- **Node.js 20.x** -- Puppeteer mit Stealth-Plugin für Bot-Simulation
- **Browser-Automatisierung** -- Playwright Chromium für Headless-Browser-Tests
- **Git-Klone** -- testssl.sh, recon-ng, spiderfoot, SecLists, waf-bypass, PayloadsAllTheThings
- **Suite-Bereitstellung** -- 19 Traffic-Suiten aus diesem Repository geklont nach `/opt/traffic-generator/suites/`
- **Konfigurationsdatei** -- `config.env` mit `TARGET_FQDN`, `TARGET_ORIGIN_IP` und `CRAPI_PORT`

Das Cloud-Init verwendet Terraform-Template-Variablen: `${target_fqdn}`, `${target_origin_ip}` und `${tool_tier}`. Shell-Variablen wie `$NPROC` werden als `$$NPROC` in der Terraform-templatefile-Funktion escaped.

```yaml file=./_data/cloud-init.yaml
```

### Ausgaben

`outputs.tf` stellt 17 Ausgaben bereit, einschließlich öffentlicher/privater IP, SSH-Befehl, Ressourcenkennungen und komponentenspezifischer Werte (target_fqdn, status_check):

```hcl file=./_data/outputs.tf
```

### Beispiel-Variablendatei

Kopieren Sie `terraform.tfvars.example` nach `terraform.tfvars` und füllen Sie Ihre Werte ein. Die `.gitignore` schließt `terraform.tfvars` aus, um das Committen von Anmeldedaten zu verhindern:

```hcl file=./_data/terraform.tfvars.example
```

Ersetzen Sie `subscription_id` durch Ihre Azure-Abonnement-ID (aus `az account show --query id -o tsv`) und `target_fqdn` durch Ihre F5 XC Load-Balancer-Domain.

## Bereitstellung

```bash
terraform init

terraform plan

terraform apply
```

Terraform gibt nach erfolgreicher Bereitstellung die öffentliche IP, den SSH-Befehl und den Ziel-FQDN aus.

## VM-Dimensionierung

Die rechenoptimierten VMs der F-Serie werden für diese CPU-intensive Traffic-Generierungs-Workload empfohlen. Das Cloud-Init-Kernel-Tuning und die Tool-Parallelität skalieren automatisch mit der vCPU-Anzahl.

| VM-SKU | vCPU | RAM | Netzwerk | Anwendungsfall |
|---|---|---|---|---|
| Standard_F4s_v2 | 4 | 8 GiB | 10 Gbps | Leichtes Labor/Demo |
| Standard_F8s_v2 | 8 | 16 GiB | 12,5 Gbps | Standard-Demos |
| Standard_F16s_v2 | 16 | 32 GiB | 12,5 Gbps | Lasttesting (Standard) |
| Standard_F32s_v2 | 32 | 64 GiB | 16 Gbps | Intensives Benchmarking |
| Standard_D16s_v3 | 16 | 64 GiB | 12,5 Gbps | RAM-intensive Workloads |

:::tip
F16s_v2 wurde durch A/B/C/D-Benchmark-Tests gegen CDN-F16s_v2- und F32s_v2-Ziele als optimale Preis-Leistungs-Option validiert. Siehe [benchmarks/](https://github.com/f5xc-salesdemos/traffic-generator/tree/main/benchmarks) für vollständige Ergebnisse.
:::

## Überprüfung nach der Bereitstellung

Die Cloud-Init-Bereitstellung dauert **15-20 Minuten für die Standard-Stufe** und **ungefähr 25 Minuten für die vollständige Stufe**. Die VM ist sofort per SSH erreichbar, aber die Tools werden noch installiert, bis cloud-init abgeschlossen ist.

### Auf Cloud-Init warten

Verbinden Sie sich per SSH mit der VM und warten Sie, bis die Bereitstellung abgeschlossen ist:

```bash
ssh azureuser@$(terraform output -raw public_ip)

cloud-init status --wait
```

Erwartete Ausgabe nach Abschluss:

```
status: done
```

### Statusdatei prüfen

Das Cloud-Init-Skript schreibt nach Abschluss eine Statusdatei:

```bash
cat /opt/traffic-generator/status.json
```

Erwartete Ausgabe:

```json
{
  "status": "ready",
  "tier": "standard",
  "target_fqdn": "demo.example.com",
  "completed": "2026-04-25T14:30:00Z",
  "tools_installed": true
}
```

### Kritische Tools überprüfen

Führen Sie eine Überprüfungsschleife aus, um zu bestätigen, dass jedes kritische Tool im PATH verfügbar ist:

```bash
for tool in nikto sqlmap nuclei dalfox ffuf gobuster feroxbuster nmap masscan \
  hydra medusa ncrack sslscan tshark hping3 mitmproxy sslyze subfinder httpx \
  amass arjun mtr whatweb dirb socat netcat ngrep iperf3 whois dig wrk hey; do
  if command -v "$tool" > /dev/null 2>&1; then
    echo "OK: $tool"
  else
    echo "MISSING: $tool"
  fi
done
```

### Suite-Runner überprüfen

Bestätigen Sie, dass der Runner und die Suiten installiert sind:

```bash
ls /opt/traffic-generator/suites/runner.sh
ls /opt/traffic-generator/suites/*/

/opt/traffic-generator/suites/runner.sh web-app-attacks --dry-run
```

## Anbindung an vorgelagerte Komponenten

Der Traffic-Generator zielt auf einen F5 XC HTTP Load Balancer, der den Origin-Server schützt. Stellen Sie zuerst den Origin-Server bereit, konfigurieren Sie F5 XC und setzen Sie dann `target_fqdn` auf den FQDN des Load Balancers:

```bash
# Nach der Bereitstellung des Origin-Servers und der Erstellung eines F5 XC HTTP Load Balancers:
cat > terraform.tfvars <<EOF
subscription_id  = "your-subscription-id"
target_fqdn      = "your-xc-load-balancer.example.com"
target_origin_ip = "$(cd ../../origin-server/terraform && terraform output -raw public_ip)"
EOF
```

| Erforderliche Variable | Quelle | Beschreibung |
|---|---|---|
| `target_fqdn` | F5 XC-Konsole | FQDN des HTTP Load Balancers, der den Origin-Server schützt |
| `target_origin_ip` | `public_ip`-Ausgabe des Origin-Servers | Direkte Origin-IP für Bypass-Tests (optional) |

Fahren Sie mit dem [Tool-Katalog](../04-tool-catalog/) für die vollständige Tool-Referenz oder den [Suiten](../05-suites/) fort, um mit der Traffic-Generierung zu beginnen.
